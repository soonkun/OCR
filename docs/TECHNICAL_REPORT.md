# 고문서 OCR 복원 — 기술 보고서

한국어+한자 혼용 저화질 고문서(예: 1962년 농촌진흥청 간행물, 2/3단 세로조판)를
디지털 텍스트로 복원하기 위한 시스템의 설계와 주요 기술 결정을 정리한다.

---

## 1. 목표와 난점

- **대상**: 흐릿한 활자, 사진·삽화 혼재, **2단·3단 세로조판**, 한국어와 한자가 섞인 옛 인쇄물.
- **난점**
  1. 단일 PaddleOCR 모델은 한글(11,172자)과 한자(15,565자)를 **동시에** 못 읽는다.
  2. 다단 조판을 통째로 OCR하면 같은 높이의 여러 칸이 뒤섞여 읽기 순서가 깨진다.
  3. 사진·삽화가 많아 단순 잉크 기반 칸 검출이 흔들린다.
  4. 한자는 문맥에 따라 독음이 달라(樂→락/악/요) 기계적 변환이 어렵다.

---

## 2. 전체 아키텍처

```
업로드 → 전처리(preprocess) → 다단 분리/듀얼패스 OCR(layout, dual_ocr, ocr_engine)
      → (한글 병기 LLM 보정, llm_correct) → 저장(db) → 웹 UI(폴링)
```

- **백엔드**: FastAPI + SQLite + `ThreadPoolExecutor` 백그라운드 처리. 정적 프론트 서빙.
- **프론트엔드**: 빌드 없는 단일 페이지(바닐라 JS). `/api/documents` 폴링으로 진행 표시.
- **데이터 흐름**: 처리는 스레드풀에서, 진행상황은 `db`를 통해 갱신, UI는 1초(처리 중)/
  4초(유휴) 간격으로 폴링.

파일별 역할은 README "아키텍처" 표 참조.

---

## 3. 처리 파이프라인 (`backend/pipeline.py`)

1. **로드**: 이미지 또는 PDF(페이지별 렌더, pypdfium2, scale≈3.0 ≒ 216DPI).
2. **전처리** (`preprocess.py`): 그레이 → 톤(밝기/대비/감마) → 디노이즈 → 샤픈 →
   업스케일(긴 변 < 1600px면 확대) → deskew → (선택)이진화.
   - **이진화 기본 OFF**: 흐린 활자에서 Sauvola/adaptive 이진화가 획을 끊어 인식률을
     크게 떨어뜨린다(실측). PP-OCRv5는 그레이/컬러 입력에서 더 잘 동작.
3. **OCR**: 혼용 모드면 듀얼패스+다단 복원, 아니면 단일 언어 OCR.
4. **LLM 보정**(혼용 + 키 있을 때): 페이지를 배치로 묶어 한글 병기로 후보정.
5. **저장**: 페이지별 미리보기(`{id}_p{N}.png`) + 결과 텍스트(페이지 구분선 포함).

처리 중 **매 페이지 시작 시 삭제 여부를 확인**해, 사용자가 삭제하면 즉시 중단한다(아래 7-D).

---

## 4. 다단(多段) 읽기순서 복원 (`backend/layout.py`)

핵심 난점 ②③에 대한 해법.

- **초기 방식(폐기)**: 열별 '잉크량' 프로파일에서 빈 세로 띠(거터)를 찾아 칸을 나눔.
  → 사진·삽화의 잉크가 거터를 메우거나, 그림 사이 흰 틈을 거터로 오인해 흔들렸다.
- **현재 방식**: **OCR이 찾은 텍스트 줄 박스의 x분포**로 단을 추정한다.
  1. 페이지 전체를 ko·han 모델로 **한 번씩** OCR해 줄 박스를 모은다(칸별 분할-OCR보다 빠름).
  2. 줄 박스의 x구간을 투영해 '글자 줄이 덮는 띠(=칸)'를 찾는다. **사진·삽화는 텍스트
     박스를 만들지 않으므로** 단 구조를 망치지 않는다.
  3. 여러 칸을 가로지르는 넓은 줄(제목)은 제외, 칸당 최소 줄 수 미만이면 단단으로 폴백.
  4. 각 줄을 칸에 배정 → **칸 좌→우, 칸 안 위→아래** 로 정렬해 읽기 순서를 복원.
- 결과: 표지=1단, 본문=2/3단을 자동 판별. 54쪽 실문서로 검증.

## 5. 한글+한자 듀얼패스 (`backend/dual_ocr.py`)

- `korean` 모델과 `chinese_cht` 모델을 같은 페이지에 돌려, **줄 박스 중심 최근접**으로
  짝지어 `{ko, han, ko_score, han_score, box}` 후보쌍을 만든다(두 패스는 동일 검출 모델을
  써 박스가 거의 일치).
- LLM이 없으면 줄마다 신뢰도 높은 후보를 택하는 **폴백 병합**(한자 병기 없음).

---

## 6. 한글 병기 LLM 후보정 (`backend/llm_correct.py`)

- 입력: 페이지별 (한글모델 텍스트, 한자모델 텍스트) 후보쌍. 토큰 절약을 위해 **여러 페이지를
  배치로** 묶어 OpenAI Chat Completions에 보낸다(스트리밍 수신).
- **충실성 우선 프롬프트**: 두 OCR 후보에 실제로 나타난 글자만 근거로 삼고, 배경지식으로
  날짜·숫자·이름을 지어내지 않는다. 판독 불가는 `□`, 문맥상 거의 확실한 한두 글자 보충은
  `〔 〕`로 표시. 1962년 당시 옛 표기를 보존하고, 한자(어)는 `한글독음(漢字)`로 병기.
- 키/SDK가 없으면 `available()=False` → 폴백 병합 사용(앱은 키 없이도 동작).

## 7. 개발 중 해결한 주요 문제

### A. PaddlePaddle 3.0.0 추론이 깨짐 → 3.1.0
- 증상: `paddle_inference.create_predictor` 에서
  `ValueError: Type of attribute: strides is not right` (PIR 다이얼렉트). OCR이 한 글자도
  안 나옴.
- 원인: paddlepaddle **3.0.0**의 CPU PIR(새 IR) 추론 버그. 모델 파일이 PIR 포맷이라
  `enable_new_ir=False`로도 우회 불가.
- 해결: **paddlepaddle 3.1.0** 으로 업그레이드. `requirements.txt`에 `>=3.1` 명시,
  3.0.0 금지 경고 추가.

### B. `enable_mkldnn=False` 가 segfault 유발
- 3.0.0용 우회로 넣었던 `enable_mkldnn=False`가 3.1.0 + 큰 이미지에서 **Segmentation
  fault**를 일으켰다. → 제거하고 MKLDNN 기본값(켜짐) 유지(`ocr_engine.py`).

### C. 다단 검출이 사진에 흔들림
- 잉크 기반 → **텍스트 줄 박스 기반**으로 재작성(4번). 사진이 거터를 덮은 3단 페이지도
  정확히 복원.

### D. 삭제해도 OCR이 안 멈춤
- 증상: UI에서 '삭제'해도 백그라운드 OCR 스레드는 끝까지 돌고(심지어 LLM API까지 호출),
  서버 프로세스를 닫아야만 멈췄다.
- 해결: 파이프라인이 **매 페이지 시작과 LLM 호출 직전에 `db.get_document`로 존재 여부를
  확인**해, 삭제되었으면 즉시 중단(불필요한 CPU·유료 API 낭비 차단).

### E. 프론트 업데이트가 캐시에 막힘
- 증상: 코드를 고쳐도 브라우저가 옛 `app.js`를 캐시해 새 기능이 안 보임.
- 해결(2중):
  1. `/app` 응답에 **`Cache-Control: no-cache, no-store`** 미들웨어(`main.py`).
  2. `index.html`이 **버전 쿼리(`app.js?v=...`)** 로 자산을 참조 → 버전만 올리면 새 URL이라
     캐시를 무조건 우회.

### F. 배치 런처가 한글 때문에 깨짐
- 증상: `.bat`에 한글이 들어가면 CP949 콘솔에서 깨져 명령으로 오인돼 에러. 브라우저 자동
  열기까지 도달하지 못함.
- 해결: 런처(`start.bat`, `고문서OCR-실행.bat`)를 **순수 ASCII**로 작성. 동작은 기존 서버
  정리(8000 포트) → 서버 시작(`--no-access-log`) → 헬스 폴링 후 브라우저 열기.

## 8. 미리보기 시스템 (`backend/preview.py`, 결과 페이지 넘김)

- **전처리 조정용**: 업로드 파일을 토큰으로 캐시(LRU). PDF는 모든 페이지를 메모리에 올리지
  않고 **요청 페이지만 그때 렌더**(원본 바이트 + 최근 렌더 캐시). `/preview/render?page=N`.
- **결과 화면용**: 처리 중 페이지별 전처리 이미지를 `{id}_p{N}.png`로 저장. 완료 후
  `/documents/{id}/preview?page=N` 으로 아무 페이지나 조회. 프론트는 페이지를 넘기면
  이미지와 결과 텍스트(페이지 구분선 기준 스크롤)를 함께 이동시켜 대조 편집을 돕는다.

---

## 9. 프론트엔드 (`frontend/`)

빌드 없는 바닐라 JS 단일 페이지. 주요 요소:
- 폴링 기반 진행바, 처리 중 현재 페이지 미리보기.
- 전처리 조정 모달(슬라이더·토글·페이지 넘김·적합도 측정).
- 결과 화면 페이지 넘김(이미지↔텍스트 동기), **드래그 구분선**(좌우 폭 조절, 미리보기
  `object-fit: contain` 폭맞춤), **글씨 크기**(zoom 변수, 높이·투어 좌표 보정), **온보딩
  투어**(스포트라이트), 모두 `localStorage`로 상태 기억.
- `[hidden]{display:none!important}` 로 클래스 `display` 가 HTML `hidden`을 덮어쓰는 버그
  방지.

## 10. 알려진 한계 / 향후 과제

- **사진·표가 칸 구조를 깨는 일부 페이지**는 단 검출이 흔들릴 수 있다(텍스트 박스 기반으로
  크게 개선했으나 100%는 아님).
- **'다시 처리'를 진행 중에 누르면** 옛 작업과 새 작업이 겹칠 수 있다(삭제 취소만 구현됨).
  세대(generation) 토큰 기반 취소가 향후 과제.
- LLM 보정은 OpenAI API 비용이 발생하며, 배치 마지막 호출은 삭제 직후 한두 번 더 나갈 수 있다.
- CPU 추론이라 페이지가 많은 문서는 처리 시간이 길다(GPU·배치 최적화 여지).
